# Drill Press Closed-Loop PID Control & Telemetry System

This repository contains the firmware and software for a closed-loop motor control system designed for a drill press. The architecture is split between an **STM32F4** microcontroller, which handles the strict real-time hardware control, and an **ESP32**, which acts as a Wi-Fi gateway to host a real-time interactive web dashboard.

---

## STM32 Architecture (Real-Time Motor Control)

The STM32 firmware is written in C using the STM32 HAL library. It is heavily optimized to survive Electromagnetic Interference (EMI) and "ground bounce" generated by the heavy inductive load of the drill press motor.

### `main.c` & `main.h`
The central nervous system of the microcontroller.
* **Initialization & Timing:** Configures all hardware peripherals including Timers (TIM1 for PWM, TIM2 for Input Capture, TIM8 for the 10ms control loop).
* **The Main Loop:** Operates a non-blocking `while(1)` loop that schedules LCD updates, checks for RPM timeouts, and dispatches telemetry to the ESP32.
* **Timer8 Callback:** Interrupt given every 10ms to call the motor feedback update function.

### `motor_control.c` & `motor_control.h`
Handles the mathematics of the closed-loop speed control.
* **Feed-Forward Baseline:** Calculates a baseline PWM duty cycle based on the target RPM, meaning the PID loop only has to correct for physical disturbances.
* **Proportional Gain:** Uses only proportional feedback control to keep actual RPM at the target RPM.

### `rpm_sensor.c` & `rpm_sensor.h`
Responsible for interpreting the raw optical encoder data.
* **DMA Capture:** Uses Hardware Timers (TIM2) and DMA to measure the time elapsed between optical encoder pulses without blocking the CPU.
* **Signal Processing Pipeline:** Passes raw RPM data through a Median-of-3 filter to instantly reject single-point EMI glitches, followed by an Exponential Moving Average (EMA) Alpha filter to smooth out quantization jitter.
* **0 RPM Timeout:** Uses a calculated timeout that determines the drill has stopped spinning if a pulse hasn't been received during the timeout duration.

### `lcd.c` & `lcd.h`
A highly robust, custom I2C driver for a 16x4 character display.
* **Non-Blocking DMA:** Formats display strings and sends them via DMA, preventing the slow I2C protocol from starving the high-speed PID control loop.
* **Soft Resets:** Executes a rapid `LCD_Soft_Reset()` before transmissions to cure minor EMI brain-damage without causing the screen to flicker.
* **Advanced Fault Recovery:** Features a state machine that handles catastrophic I2C bus crashes. It utilizes a 2-second cooldown timer to prevent interrupt storms, kills "Zombie DMA" transfers, and can execute a "9-Clock Pulse" (`I2C_ClearBus`) to force the LCD chip to release the SDA data line if it holds the bus hostage.

### `telemetry.c` & `telemetry.h`
Manages asynchronous communication via UART.
* **ESP32 Transmission:** Packs the current RPM and PWM into a compact C-struct (`TelemetryPacket_t`) and blasts it to the ESP32 via UART DMA.
* **ESP32 Reception:** Decodes strings sent from the ESP32 to implement new target RPM values entered on the website.

## ESP32 Architecture (Web Server & Dashboard)

The ESP32 acts as an asynchronous web server, bridging the gap between the STM32's raw serial data and a rich, graphical user interface accessible from any browser.

### `RPM-vs-Time-Graph-ESP32.ino`
The core Arduino sketch running on the ESP32.
* **Network Setup:** Establishes a Wi-Fi Access Point to serve the web dashboard.
* **Asynchronous Web Server:** Uses `ESPAsyncWebServer` to host the HTML files without blocking the main loop.
* **UART to Web Pipeline:** Reads the incoming UART serial stream from the STM32, parses the RPM and PWM targets, and pushes them instantly to connected web clients using Server-Sent Events (SSE) for ultra-low latency updates. 
* **TFT LCD Integrated Display:** The parsed RPM and PWM information is also used to update a built-in TFT LCD display on the ESP32 for an additional graphical display.
* **Command Routing:** Receives HTTP GET requests from the web interface (when the user types in a new target RPM) and translates them back into UART commands to send down to the STM32.

### `indexhtml.h`
The frontend User Interface, stored in PROGMEM.
* **Live Dashboard:** A modern UI built with HTML, CSS, and vanilla JavaScript that displays graphs and values for the current actual RPM, target RPM, RPM error, and applied PWM over time.
* **Edge Compute:** Minimizes transmit package size from the STM32 by calculating the Live RPM Error (Target - Actual) directly in the browser.

### `chartjs.h`
* **Offline Rendering:** Contains the minified source code for the `Chart.js` library. By hosting this directly on the ESP32, the entire dashboard and graphing system works perfectly offline in a garage or workshop without requiring an active internet connection.

### `flappy.h` & `pacman.h`
* **Arcade Easter Eggs:** HTML/JS payloads for embedded "break time" arcade games. These are cleverly hidden behind an interactive menu on the dashboard, proving that the asynchronous web server can handle heavy UI elements without interrupting the critical background SSE telemetry stream.

> **Note:** To implement the arcade easter eggs, code needs to be uncommented in both the `RPM-vs-Time-Graph-ESP32.ino` file and `indexhtml.h` file everywhere there is a "TODO - arcade" block, and then reflash the ESP32 with the new uncommented code.
